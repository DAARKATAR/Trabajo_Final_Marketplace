1. DIAGRAMA DE ARQUITECTURA DEL SISTEMA
--------------------------------------------------------------------------------
La topología del sistema representa una arquitectura descentralizada de microservicios con un API Gateway centralizador, un directorio de servicios desacoplado (UDDI) y canales híbridos de comunicación que combinan REST/JSON para servicios internos modernos y SOAP/XML para la integración con sistemas bancarios legacy (heredados). Cada microservicio de negocio (Usuarios, Productos, Órdenes, Pagos) opera con su propia base de datos lógica conectada a un servidor común de PostgreSQL, garantizando el aislamiento de datos (Database-per-Service pattern).

Descripción del flujo de datos:
1. El cliente inicia una transacción enviando un payload REST/JSON al API Gateway (puerto 3000).
2. El API Gateway consulta el listado de servicios UDDI (servicios.json) para resolver dinámicamente la IP y puerto del microservicio destino.
3. Al recibir una orden de compra, el microservicio de Órdenes (puerto 3003) realiza llamadas internas (GET) a los microservicios de Usuarios y Productos para validar datos corporativos.
4. Los microservicios de Usuarios y Productos consultan y validan la información directamente en sus bases de datos lógicas aisladas ("usuarios_db" y "productos_db") en PostgreSQL.
5. El microservicio de Órdenes procesa el pago enviando un XML al banco mock SOAP (puerto 3004).
6. El microservicio de Pagos registra la transacción bancaria de forma aislada en su propia base de datos lógica "pagos_db" en PostgreSQL.
7. Una vez que el banco retorna la aprobación en XML, el microservicio de Órdenes guarda la transacción en su base de datos lógica "ordenes_db" y retorna el estado de confirmación en JSON.


2. EXPLICACIÓN DE CADA MICROSERVICIO Y BASES DE DATOS LÓGICAS
--------------------------------------------------------------------------------
Cada componente del sistema se ha diseñado bajo los principios de desacoplamiento y responsabilidad única, respetando la estructura de carpetas independiente y sus propias conexiones de base de datos.

A. API Gateway (Carpeta: /api-gateway)
- Rol: Punto de entrada único (Proxy Reverso) expuesto en el puerto 3000.
- Responsabilidad: Enrutar de manera transparente las peticiones externas hacia las APIs internas. Centraliza la resolución de nombres resolviendo las URLs de destino a través de un JSON local que representa el registro UDDI.
- Persistencia: No requiere base de datos (Stateless).

B. Microservicio de Usuarios (Carpeta: /usuarios-service)
- Rol: Gestión del ciclo de vida del usuario (puerto 3001).
- Responsabilidad: Permite el registro, consulta, modificación y eliminación de usuarios (CRUD).
- Conexión de Base de Datos: Se conecta a la base de datos lógica de PostgreSQL "usuarios_db" usando la librería de Node.js 'pg' (Pool de conexiones configurado en /usuarios-service/db.js).
- Tabla en Postgres:
  CREATE TABLE usuarios (
      id SERIAL PRIMARY KEY,
      nombre VARCHAR(100) NOT NULL,
      correo VARCHAR(100) UNIQUE NOT NULL,
      contrasena VARCHAR(100) NOT NULL DEFAULT '123456'
  );

C. Microservicio de Productos (Carpeta: /productos-service)
- Rol: Catálogo e inventario de productos (puerto 3002).
- Responsabilidad: Mantenimiento del catálogo de artículos disponibles en el Marketplace (ID, nombre, precio base). El CRUD de creación y edición está restringido exclusivamente a usuarios con rol de Administrador.
- Conexión de Base de Datos: Conexión aislada a la base de datos lógica de PostgreSQL "productos_db" configurada en /productos-service/db.js.
- Tabla en Postgres:
  CREATE TABLE productos (
      id SERIAL PRIMARY KEY,
      nombre VARCHAR(100) NOT NULL,
      precio DECIMAL(10, 2) NOT NULL
  );

D. Microservicio de Órdenes (Carpeta: /ordenes-service)
- Rol: Orquestación del flujo de compras (puerto 3003).
- Responsabilidad: Valida la existencia del cliente, la disponibilidad del producto en sus servicios correspondientes, calcula el valor total de la compra, solicita el pago seguro al banco por SOAP/XML, y genera el registro de la transacción.
- Conexión de Base de Datos: Conexión exclusiva a la base de datos lógica de PostgreSQL "ordenes_db" configurada en /ordenes-service/db.js.
- Tabla en Postgres:
  CREATE TABLE ordenes (
      id SERIAL PRIMARY KEY,
      usuario_id INTEGER NOT NULL,
      producto_id INTEGER NOT NULL,
      cantidad INTEGER NOT NULL,
      total DECIMAL(10, 2) NOT NULL,
      estado_pago VARCHAR(50) NOT NULL,
      fecha_creacion TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
      codigo_compra VARCHAR(100),
      productos_detalle TEXT
  );

E. Microservicio de Pagos SOAP/XML (Carpeta: /pagos-soap-service)
- Rol: Banco Mock Legacy expuesto en el puerto 3004.
- Responsabilidad: Simula el core bancario corporativo tradicional que recibe transferencias financieras formateadas exclusivamente en XML a través de una petición HTTP POST en la ruta `/pagos`.
- Conexión de Base de Datos: Conexión exclusiva a la base de datos lógica de PostgreSQL "pagos_db" configurada en /pagos-soap-service/db.js para persistir el historial de transacciones financieras procesadas.
- Tabla en Postgres:
  CREATE TABLE pagos (
      id SERIAL PRIMARY KEY,
      usuario_id INTEGER NOT NULL,
      monto DECIMAL(10, 2) NOT NULL,
      estado VARCHAR(50) NOT NULL,
      fecha_procesamiento TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  );

F. Registro UDDI (Carpeta: /registro-servicios)
- Rol: Service Directory o Páginas Amarillas del sistema.
- Responsabilidad: Almacenar los metadatos de configuración de red de todos los servicios activos en el archivo de texto estructurado `servicios.json`. Evita que las rutas internas de los servicios se declaren estáticas en el código del Gateway.
- Persistencia: No requiere conexión de base de datos activa (Stateless - Lee archivo JSON).


3. EJEMPLOS DE FORMATO JSON USADO
--------------------------------------------------------------------------------
El formato JSON se utiliza para toda la comunicación interna REST del Marketplace.

A. Creación de Usuario (Petición)
{
    "nombre": "Juan Pérez",
    "correo": "juan.perez@email.com"
}

B. Creación de Usuario (Respuesta del servicio de Usuarios)
{
    "id": 2,
    "nombre": "Juan Pérez",
    "correo": "juan.perez@email.com"
}

C. Solicitud de Compra (Petición recibida por el Gateway / Órdenes)
{
    "usuarioId": 1,
    "productos": [
        { "productoId": 1, "cantidad": 1 },
        { "productoId": 3, "cantidad": 2 }
    ]
}
